Java Collection Classes for Parallel, Distributed and on-Disk Computations

What is PDQCollection?

PDQCollection are Java collection classes (Map, Set, etc.) that are specially designed for parallel and distributed data-intensive programming.

PDQCollection is a unique programming model that: (a) operates on associative containers, (b) executes in shared-memory parallel mode as well as distributed, (c) supports data sizes that scale from in-memory to disk-backed, and (d) allows developing parallel/distributed programs in the natural sequential and object-oriented style of programming.

How does it work?

The key insight to PDQCollection is the Split-Replication-Merge computation model. The input is first splitted into smaller problems and the output is replicated into containers to hold partial results. Parallel or distributed threads compute your function for each pair of partial input and partial output. The partial results are then merged to the final output.

The second insight is that the parallelism is "built in" into the data structures, rather than scattered over your computation code. As a programmer, you only need to define how two values are to be added, and the library will take care of the rest. See the Word Count example for this concept in action.

Is it any good?

Oh, we'll say! And here's why:

  • PDQCollection does not use locks, ever!
    • Concurrent data structures (e.g. ConcurrentHashMap) use locks to protect concurrent modifications. For data-intensive problems, these locks are contests so often that the performance with multiple threads turns out to be actually poorer than sequential!
  • PDQCollection does not cache data.
    • Disk-backed structures are two-tiered: in-memory cache and on-disk B-tree. The cache miss penalty increases as the cache ratio gets smaller. PDQCollection library, however, reads and writes to disk only once, and does not require caches. By avoiding caching and random disk accesses, the PDQCollection classes significantly outperforms traditional disk-backed data structures.
  • PDQCollection does not randomly seek data on disk or remote machines.
    • PDQCollection classes always access the out-of-core data (on disk or remote machine) as a sequential stream of bytes, for reading and writing.

Please refer to the Technical Report for detailed performance analysis with concurrent data structures, BerkeleyDB and Hazelcast distributed map.

Where can it be used?

The Technical Report gives several examples to illustrate the capabilities of PDQCollections library with several applications drawn from a variety of fields in Computer Science, including

  • Machine Learning (see the KMeans example for clustering)
  • Data Mining (see the FP Growth example for frequent itemset discovery)
  • Graph Processing (see the Pagerank example)
  • Relational processing of structured data
  • Incremental processing of log data
